> **发表时间**：2026年3月1日

## 一、研究背景与核心问题

### 1.1 从传统RAG到Agentic MM-RAG的范式转变

多模态大语言模型（MLLMs）正在经历从简单问答向复杂推理的跨越式发展。传统的检索增强生成（RAG）范式采用固定的"检索-生成"流水线，在文本-only场景中已证明其价值。然而，随着MLLMs能力的快速提升，业界对模型的期望也从简单的知识检索转向**自适应的多步推理、跨模态证据整合和知识密集型查询处理**。

这一转变催生了一个新兴领域——**多模态Agentic RAG（MM-RAG）**。与传统RAG不同，Agentic MM-RAG要求模型具备：

- **迭代任务分解**：将复杂查询拆解为可管理的子问题
- **自适应跨模态检索**：根据推理需求动态选择文本或图像证据
- **多模态证据整合**：将来自不同模态的信息融合为连贯的答案
- **长推理链能力**：支持4-5步甚至更多步骤的深度推理

### 1.2 现有基准的局限性

尽管已有多个MM-RAG基准（如OK-VQA、ViQuAE、WebQA、InfoSeek、MMSearch等），但它们普遍存在以下问题：

| 局限性 | 具体表现 | 影响 |
|--------|----------|------|
| **浅层推理** | 大多数基准仅支持1-2步检索 | 无法评估长推理链能力 |
| **固定流水线** | 采用固定的retrieve-then-generate模式 | 无法测试自适应规划能力 |
| **模态单一化** | 将多模态证据压缩为文本通道 | 无法评估真正的跨模态推理 |
| **缺乏过程标注** | 没有逐步推理的详细标注 | 无法进行细粒度的错误分析 |
| **无推理拓扑** | 未定义推理结构的多样性 | 无法系统评估不同推理模式 |

这些局限性使得现有基准难以真实反映MLLMs在复杂多模态场景下的推理能力。

### 1.3 MC-Search的核心使命

MC-Search（Multimodal agentic RAG with long, structured Chains of Search-enhanced reasoning）旨在填补上述空白，其核心使命包括：

1. **构建首个面向Agentic MM-RAG的长推理链基准**
2. **定义五种代表性推理拓扑结构**
3. **提供细粒度的过程级评估指标**
4. **开发过程级对齐框架Search-Align**

## 二、MC-Search基准详解

### 2.1 五大推理拓扑结构

MC-Search的核心创新之一是定义了五种代表性推理拓扑，覆盖串行和并行两种基本推理模式：

#### （1）Image-Initiated Chain（图像起始链）

**结构特征**：
- 第一步从图像检索开始
- 后续步骤基于文本检索进行验证和扩展
- 适用于图像包含查询和文本补充知识的场景

**典型示例**：
```
问题：图中描绘了什么圣经场景，这幅画目前收藏于何处，
      何时以何价格被收购，这对艺术家在美国博物馆史上的意义是什么？

推理链：
1. [图像] 图中描绘了什么圣经场景？ → 圣母领报（The Annunciation）
2. [文本] 这幅画收藏于哪个博物馆？ → 费城艺术博物馆
3. [文本] 何时以何价格被收购？ → 1899年4月5日，1750美元
4. [文本] 这对艺术家在美国博物馆史上的意义？ → 这是Tanner第一件被美国博物馆购买的作品
```

**分布占比**：1,306个样本（39.2%）

#### （2）Text-Initiated Chain（文本起始链）

**结构特征**：
- 第一步从文本检索开始获取事实知识
- 后续步骤引入图像检索进行视觉验证
- 适用于文本知识为主、视觉确认的场景

**典型示例**：
```
问题：图像是否视觉上确认了文本中描述的Solvang独特文化特征？

推理链：
1. [文本] 文本如何描述Solvang的文化特征？ → "加州中一片丹麦风情"
2. [图像] 图像是否视觉上确认了这种文化特征？ → 是，显示了风车建筑和欧式风格
```

**分布占比**：169个样本（5.1%）

#### （3）Parallel Image-Text Fork（并行图文分叉）

**结构特征**：
- 图像和文本检索并行进行，无严格依赖关系
- 两个分支的结果需要跨模态协调整合
- 是最具挑战性的推理拓扑之一

**典型示例**：
```
问题：根据显示的曼陀林外观，Pasquale Vinaccia 1835年的弦材料创新
      如何导致乐器结构变化及其对曼陀林设计的持久影响？

推理链：
1. [图像] 曼陀林上可见什么类型的弦？ → 钢丝弦
2. [文本] 这些新钢丝弦需要对曼陀林琴体做什么结构修改？ → 需要加固琴体、加深共鸣箱
3. [文本] 加深共鸣箱增强了什么声学品质？ → 增强了音调共鸣
4. [文本] Vinaccia使用钢丝弦的决定对曼陀林设计产生了什么长期影响？ → 成为主流的弦乐方式
```

**分布占比**：680个样本（20.4%）

#### （4）Multi-Images Fork（多图像分叉）

**结构特征**：
- 需要检索多个图像进行视觉比较
- 后续步骤引入文本证据进行事实支持
- 适用于多图像对比和视觉推理场景

**典型示例**：
```
问题：Epcot的加拿大馆和摩洛哥馆中，哪个以高耸纤细的塔为主要建筑特色，
      其作为世界展示首个扩建展馆的历史意义是什么？

推理链：
1. [图像1] 加拿大馆屋顶最显著的特征是什么？ → 非常陡峭的多层绿色屋顶
2. [图像2] 摩洛哥馆天际线最显著的建筑特征是什么？ → 高耸的矩形塔楼
3. [图像] 对比两个展馆，哪个以高塔而非陡峭屋顶为主要特征？ → 摩洛哥馆
4. [文本] 摩洛哥馆作为世界展示扩建的历史意义是什么？ → 1984年9月7日开放的第一个扩建展馆
```

**分布占比**：233个样本（7.0%）

#### （5）Text-Only Chain（纯文本链）

**结构特征**：
- 完全通过文本检索进行推理
- 作为对比基线，验证多模态带来的增益
- 适用于纯文本知识密集型查询

**典型示例**：
```
问题：Bentley Mulsanne的关键属性是什么，包括生产周期、发动机规格、
      命名来源和著名特别版？

推理链：
1. [文本] Bentley Mulsanne是什么类型的车，何时生产，命名来源？ 
   → 2010-2020年生产的全尺寸豪华车，以勒芒赛道的Mulsanne弯道命名
2. [文本] Bentley Mulsanne的发动机规格是什么？ 
   → 6.75升Bentley L系列V8发动机
3. [文本] 有什么著名特别版？ 
   → "W.O. Edition"，扶手处展示W.O. Bentley个人汽车的曲轴碎片
```

**分布占比**：945个样本（28.3%）

### 2.2 HAVE：逐跳归因与证据验证

构建高质量多跳推理数据集面临的核心挑战是**幻觉步骤**（plausible but ungrounded hops）和**冗余步骤**（redundant steps）。MC-Search提出了**HAVE（Hop-wise Attribution and Verification of Evidence）**机制来解决这些问题。

#### 2.2.1 上下文效用计算

对于推理链中的每一步证据 $r_t$，其上下文效用定义为移除该步骤后答案准确率的下降：

$$
\text{Util}(t) = \text{F1}(\mathcal{C}) - \text{F1}(\mathcal{C} \setminus r_t)
$$

其中 $\mathcal{C} = \{r_1, r_2, ..., r_T\}$ 是所有检索证据的集合。

#### 2.2.2 导航角色评估

某些步骤可能不直接影响答案准确率，但在连接子问题和引导后续推理中起关键作用。HAVE通过检查中间答案中的实体是否出现在下游子问题中来评估导航角色：

$$
\text{Nav}(t) = \begin{cases} 
1, & \text{if } \text{Ent}(a_t) \cap \text{Ent}(q_{t+1:T}) \neq \emptyset \\
0, & \text{otherwise}
\end{cases}
$$

#### 2.2.3 冗余步骤识别

当满足以下条件时，步骤被标记为冗余：
- 上下文效用低于阈值：$\text{Util}(t) < \text{threshold}$
- 无导航角色：$\text{Nav}(t) = 0$

#### 2.2.4 HAVE的应用流程

1. **初始生成**：使用Gemini-2.5-Flash生成约21k个样本
2. **初步过滤**：使用Qwen2.5-VL-7B过滤掉包含超过2个冗余步骤的链，剩余4.8k个候选
3. **精炼验证**：使用Gemini-2.5-Pro进行跳数缩减和样本验证
4. **HAVE验证**：使用Gemini-2.5-Flash进行最终冗余检查
5. **最终数据集**：3,333个高质量样本

## 三、评估协议：超越答案准确率

### 3.1 传统指标的局限性

仅测量最终答案准确率存在明显不足：
- 无法定位错误来源（检索？推理？规划？）
- 长推理链中，中间步骤错误可能被后续步骤补偿
- 无法评估推理过程的忠实度

### 3.2 答案级评估指标

MC-Search采用四个互补的答案级指标：

#### （1）F1 Score
标准的token级重叠分数，衡量生成答案与标准答案的文本相似度。

#### （2）ΔF1（增量F1）
衡量Agentic MM-RAG相对于模型参数知识的性能增益：

$$
\Delta\text{F1} = \text{F1} - \text{F1}_{\text{w/o } \mathcal{R}}
$$

这个指标特别重要，因为它区分了模型"知道"的内容和通过检索"学到"的内容。

#### （3）Golden F1（黄金F1）
上界指标，当模型被提供黄金推理链和黄金检索内容时达到的性能，反映任务的理论最优解。

#### （4）LLM-as-a-Judge（LJ）
使用Gemini-2.5-Pro评估预测推理链相对于标准链的质量，沿四个维度评分（0-5分）：
- **答案准确率**：与标准答案的事实一致性
- **推理连贯性**：推理步骤的逻辑流畅性
- **关键知识实体覆盖度**：是否包含所有必要实体
- **步骤对齐**：是否与标准推理链对齐

### 3.3 过程级评估指标

MC-Search的核心创新在于提出了**过程级指标**，实现细粒度的错误归因：

#### （1）Hit per Step（HPS，每步命中率）
衡量预测推理链与黄金推理链的步级证据匹配程度：

$$
\text{HPS}(\hat{\mathcal{G}}, \mathcal{G}) = \frac{1}{|\mathcal{G}|} \cdot |\{(t, t') \mid r_t \in \mathcal{G}, \hat{r}_{t'} \in \hat{\mathcal{G}}, \hat{r}_{t'} = r_t\}|
$$

通过最大权重二分图匹配处理并行推理步骤的对齐问题。

#### （2）Rollout Deviation（RD，展开偏差）
衡量预测推理链与黄金推理链的步数差异：

$$
\text{RD}(\hat{\mathcal{G}}, \mathcal{G}) = | |\hat{\mathcal{G}}| - |\mathcal{G}| |
$$

- RD = 0：推理复杂度完美对齐
- RD > 0：过度检索（over-retrieval）
- RD < 0：检索不足（under-retrieval）

## 四、Agentic MM-RAG统一流水线

### 4.1 流水线架构

MC-Search提出了一个统一的Agentic MM-RAG流水线，包含三个核心模块：

```
┌─────────────────────────────────────────────────────────────┐
│                    Agentic MM-RAG Pipeline                   │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────────┐    ┌──────────────────┐              │
│  │ 子查询与行动生成  │───→│    证据获取      │              │
│  │ (Sub-query &     │    │ (Evidence        │              │
│  │  Action Gen)     │    │  Acquisition)   │              │
│  └──────────────────┘    └────────┬─────────┘              │
│           ↑                       │                         │
│           │                       ↓                         │
│  ┌──────────────────┐    ┌──────────────────┐              │
│  │ 迭代推理与综合   │←───│    证据整合      │              │
│  │ (Iterative       │    │ (Reasoning &    │              │
│  │  Reasoning)      │    │  Synthesis)     │              │
│  └──────────────────┘    └──────────────────┘              │
└─────────────────────────────────────────────────────────────┘
```

#### 模块1：子查询与行动生成
- 将复杂问题分解为子目标
- 生成聚焦的子查询
- 自适应选择三种检索行动之一：
  - (a) 使用文本查询进行文本搜索
  - (b) 使用文本查询进行图像搜索
  - (c) 使用输入图像进行图像搜索

#### 模块2：证据获取
- 子查询通过多模态编码器嵌入
- 在对应模态的知识库中进行稠密检索
- 返回Top-1最相关的证据

#### 模块3：迭代推理与综合
- 将子答案和证据反馈给模型
- 动态检查累积证据是否充分
- 如不充分则继续下一子查询；如充分则输出最终答案

### 4.2 Search-Align：过程级对齐框架

除了评估，MC-Search还提供了改进模型能力的训练框架**Search-Align**。

#### 4.2.1 训练数据构建

1. **推理思维生成**：使用Gemini-2.5-Flash为每个推理图生成显式推理思维
2. **对话式轨迹转换**：将步骤级标注转换为连贯的对话轨迹
3. **过程级监督**：包含子问题、检索行动、证据和中间答案的完整监督

#### 4.2.2 监督微调（SFT）

在对话式轨迹上对MLLMs进行微调，使模型学习：
- 如何规划推理步骤
- 如何选择检索模态
- 如何跨步骤整合证据

#### 4.2.3 Search-Align的训练效果

| 模型 | F1提升 | HPS提升 | RD降低 |
|------|--------|---------|--------|
| InternVL3.5-8B | +2.8 | +12.0 | -0.6 |
| Qwen2.5-VL-7B | +13.7 | +16.0 | -3.1 |

Qwen2.5-VL-7B在Search-Align后，在文本中心链上几乎追平Gemini-2.5-Pro。

## 五、实验结果深度分析

### 5.1 主要发现（RQ1）

#### 5.1.1 骨干模型性能

| 模型类型 | 模型 | Image-Initiated F1 | Multi-Images F1 | Text-Initiated F1 | Parallel Fork F1 | Text Chain F1 |
|----------|------|-------------------|-----------------|-------------------|------------------|---------------|
| 闭源 | GPT-4o-Mini | 34.18 | 19.16 | 21.65 | 21.65 | 30.94 |
| 闭源 | Gemini-2.5-Flash | 37.38 | 31.89 | 27.94 | 27.94 | 33.96 |
| 闭源 | Gemini-2.5-Pro | **42.76** | **36.58** | **34.19** | **34.19** | **34.46** |
| 闭源 | Claude-3.7-Sonnet | 33.09 | 27.85 | 24.65 | 24.65 | 26.75 |
| 开源 | InternVL3.5-8B | 29.49 | 20.98 | 25.48 | 25.48 | 27.84 |
| 开源+Align | InternVL+Search-Align | 32.65 | 24.43 | 28.01 | 28.01 | 31.47 |
| 开源 | Qwen2.5-VL-7B | 8.65 | 7.53 | 14.34 | 14.34 | 14.89 |
| 开源+Align | Qwen+Search-Align | 28.05 | 19.41 | 27.94 | 27.94 | 28.89 |

**关键发现**：
- Gemini-2.5-Pro在大多数推理拓扑上表现最佳
- InternVL3.5-8B是最强的开源骨干
- Search-Align显著提升开源模型性能，特别是Qwen2.5-VL-7B（+13.7 F1）

#### 5.1.2 推理拓扑难度排序

根据所有模型的平均性能，推理拓扑难度排序为：

**Parallel Image-Text Fork（最难）** > Multi-Images Fork > Image-Initiated Chain > Text-Initiated Chain > Text Chain

**Parallel Image-Text Fork的困难点**：
- 需要同时规划和覆盖文本和图像两个分支
- 错过任一分支都会直接降低HPS
- Top-1检索约束下难以捕捉完整证据集
- 无严格顺序的并行证据对齐增加不一致风险

**Multi-Images Fork的困难点**：
- 需要将多个视觉证据与对应文本事实对齐
- 模型经常检索到正确的图像但错误归属或无法连贯整合

### 5.2 推理链长度影响（RQ2）

随着推理链长度增加，所有模型的性能都呈现**一致下降**趋势：

| 推理步数 | Gemini-2.5-Pro F1 | InternVL3.5-8B F1 | Qwen2.5-VL-7B F1 |
|----------|-------------------|-------------------|------------------|
| 1-2步 | ~45 | ~35 | ~20 |
| 3步 | ~42 | ~32 | ~15 |
| 4步 | ~35 | ~25 | ~10 |
| 5步 | ~28 | ~18 | ~8 |

**关键洞察**：
- 短推理链（1-3步）：模型间性能差异适中
- 长推理链（4-5步）：所有模型性能急剧下降
- Gemini-2.5-Pro在长步数上最稳健
- **核心挑战**：扩展多步推理链中的复合检索错误和不稳定规划

### 5.3 过度检索与检索不足（RQ3）

#### 5.3.1 现象描述

- **检索不足（Under-retrieval）**：$\Delta\text{Step} < -2$，跳过关键证据，造成无法恢复的信息缺口
- **适度过度检索（Moderate over-retrieval）**：$\Delta\text{Step} = 1-2$，通常提升准确率，额外步骤有助于从不完美规划中恢复
- **严重过度检索（Severe over-retrieval）**：$\Delta\text{Step} \geq 4$，噪声和不相关上下文导致性能急剧下降

#### 5.3.2 不同模型的检索校准能力

| 模型 | 最优ΔStep | 对过度检索的容忍度 |
|------|-----------|-------------------|
| Gemini-2.5-Pro | 0-1 | 高 |
| Gemini-2.5-Flash | 0-1 | 中等 |
| InternVL3.5-8B | 0-1 | 中等 |
| Claude-3.7-Sonnet | 0 | 低 |
| GPT-4o-Mini | 0 | 低 |

**核心张力**：在获取充分证据与避免过度检索之间存在根本性权衡。

### 5.4 模态覆盖分析（RQ4）

**关键发现**：
- 所有模型都保持强大的文本覆盖率（>78%）
- 图像覆盖率高度依赖显式图像输入提示
- Gemini-2.5-Pro在含图像查询中图像覆盖率达87%，但在纯文本查询中降至29.5%

## 八、个人理解与行业影响分析

### 8.1 技术创新点评

#### 8.1.1 对评估范式的革新

MC-Search最重要的贡献在于**将MM-RAG评估从结果导向转向过程导向**。传统的F1和准确率只能告诉你"模型答对了没有"，而HPS和RD能告诉你"模型在哪里出了问题"。这种细粒度诊断能力对模型开发者极其宝贵。

#### 8.1.2 推理拓扑的系统化

五种推理拓扑的定义不仅是分类学的贡献，更是**为后续研究提供了标准化的测试矩阵**。研究者可以针对性地优化特定拓扑（如最难的Parallel Image-Text Fork），而不是在混合数据集上盲目实验。

#### 8.1.3 HAVE数据质量控制机制

HAVE机制展示了如何用**自动化+验证**的方式构建高质量推理数据集。逐跳验证的思想可以推广到其他多跳推理场景（如知识图谱问答、科学推理等）。

### 8.2 对行业的深远影响

#### 8.2.1 企业级RAG系统优化

对于正在构建企业级多模态RAG系统的公司，MC-Search提供了：
- **诊断工具**：通过HPS和RD定位系统瓶颈

- **基准测试**：在3,333个高质量样本上系统评估迭代优化效果

#### 8.2.2 开源模型追赶闭源模型的路径

Search-Align的实验结果表明，**开源模型与闭源模型的差距主要在检索规划和步骤级推理对齐，而非基础感知或语义理解能力**。这为开源社区指明了追赶方向：
- 构建更多高质量的Agentic推理数据
- 开发过程级监督训练框架
- 专注模态感知检索能力的提升

#### 8.2.3 多模态Agent的标准化

MC-Search为**多模态AI Agent的评估**设立了新标准。随着AI Agent市场的快速发展，统一的评估基准将：
