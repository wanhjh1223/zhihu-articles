> **论文标题**: ReSearch: Learning to Reason with Search for LLMs via Reinforcement Learning  
> **发表时间**: 2025年3月27日  
> **论文链接**: [arXiv:2503.19470](https://arxiv.org/abs/2503.19470)

## 一、研究背景与核心问题

### 1.1 研究背景

大型语言模型（LLMs）在推理能力方面取得了显著进展，以OpenAI的o1和DeepSeek的R1为代表的新一代推理模型展现了强大的链式思维能力。然而，**将推理能力与外部搜索过程有效整合**仍然是一个重大挑战，尤其是在需要多步检索的复杂多跳问答场景中。

传统RAG（检索增强生成）方法存在以下局限：
- **分离式架构**：推理与检索模块相互独立，缺乏深度融合
- **人工规则依赖**：需要预设何时检索、如何检索的启发式规则
- **数据成本高**：需要大量标注数据来训练检索决策模块
- **缺乏自适应**：无法根据问题复杂度动态调整检索策略

### 1.2 核心问题

ReSearch试图解决的核心问题是：

> **如何让大模型自主学会在推理过程中何时进行搜索、如何构建搜索查询、如何利用搜索结果进行下一步推理，而无需人工标注的中间推理步骤？**

这一问题的难点在于：
1. **搜索与推理的耦合**：搜索操作应该作为推理链的一部分，而非独立模块
2. **端到端学习**：避免对推理步骤的人工标注，实现从问题到答案的直接学习
3. **涌现能力激发**：通过强化学习自然激发反思、自我修正等高级推理能力

## 二、技术方法详解

### 2.1 核心创新：搜索作为推理链的一部分

ReSearch的核心创新是将**搜索操作视为推理链的有机组成部分**，而非外部工具调用。具体来说：

#### 2.1.1 链式推理结构

ReSearch的推理链包含三种元素：

```

思考内容（文本推理）
</think>

<search>
搜索查询
</search>

<result>
检索结果
</result>
```

这种结构使得：
- **文本推理引导搜索**：模型通过思考决定何时需要搜索、搜索什么
- **搜索结果影响推理**：检索结果会反馈到后续的思考过程中
- **形成闭环**：搜索与推理相互促进，而非单向依赖

#### 2.1.2 与传统RAG的本质区别

| 维度 | 传统RAG | ReSearch |
|------|---------|----------|
| 架构 | 检索→生成 流水线 | 检索嵌入推理链 |
| 决策 | 人工规则/独立模块 | 模型自主学习 |
| 训练 | 需要检索标注 | 仅需答案标注 |
| 适应性 | 固定策略 | 动态调整 |
| 涌现能力 | 无 | 反思、自我修正 |

### 2.2 强化学习训练框架

#### 2.2.1 算法选择：GRPO

ReSearch采用**群体相对策略优化（Group Relative Policy Optimization, GRPO）**作为训练算法。GRPO是DeepSeek-R1使用的核心算法，具有以下特点：

- **无需价值模型**：通过组内相对评分估计优势，降低训练复杂度
- **样本效率高**：每次生成多个候选答案，利用组内对比学习
- **稳定性好**：相对奖励减少了绝对评分的噪声影响

#### 2.2.2 奖励设计

ReSearch采用极简的奖励机制：

**1. 答案准确性奖励（主要）**
```python
reward = F1_score(predicted_answer, gold_answer)
```

**2. 格式规范性奖励（辅助）**
- 正确使用 `<think>`、`<search>`、`<result>` 标签
- 结构完整，逻辑清晰

**关键设计**：训练中**屏蔽检索结果对损失的干扰**，避免模型过拟合训练时的特定检索结果，确保泛化能力。

#### 2.2.3 训练流程

```
输入：问题 Q
      ↓
模型生成：推理链 C（包含思考+搜索+结果）
      ↓
执行搜索：将 <search> 内容提交检索系统
      ↓
填充结果：将检索结果填入 <result> 标签
      ↓
生成答案：基于完整推理链生成最终答案 A
      ↓
计算奖励：R = F1(A, A_gold)
      ↓
GRPO更新：基于组内相对奖励更新策略
```

### 2.3 关键实现细节

#### 2.3.1 检索系统接口

ReSearch将检索系统作为环境的一部分：
- 模型输出 `<search>查询</search>`
- 系统自动执行检索并填充 `<result>检索结果</result>`
- 检索过程对模型是"黑盒"，模型只需关注何时搜索、搜索什么

#### 2.3.2 推理链生成策略

在训练过程中，模型需要学习：
1. **初始化思考**：分析问题的关键信息
2. **判断搜索时机**：何时需要外部信息
3. **构建搜索查询**：提取有效关键词
4. **整合搜索结果**：将新信息融入推理
5. **迭代或终止**：决定继续搜索还是给出答案

#### 2.3.3 长文本处理

由于推理链可能包含多轮搜索，ReSearch需要处理较长的上下文：
- 使用滑动窗口机制管理历史
- 优先保留关键推理步骤
- 动态压缩冗余信息

## 三、实验结果与性能分析

### 3.1 实验设置

**基础模型**：
- Qwen2.5-7B(-Instruct)
- Qwen2.5-32B(-Instruct)

**训练数据**：
- 仅在HotpotQA训练集上训练
- 无需中间推理步骤标注
- 仅使用问题-答案对

**评估基准**：
| 数据集 | 任务类型 | 复杂度 |
|--------|----------|--------|
| HotpotQA | 多跳问答 | 2跳推理 |
| 2WikiMultiHopQA | 多跳问答 | 多跳推理 |
| MuSiQue | 多跳问答 | 复杂多跳 |
| Bamboogle | 多跳问答 | 多样化 |

**评估指标**：
- LLM-as-a-Judge（GPT-4评分）
- 精确匹配（Exact Match）
- F1分数

### 3.2 主要实验结果

#### 3.2.1 整体性能

| 方法 | HotpotQA | 2Wiki | MuSiQue | Bamboogle |
|------|----------|-------|---------|-----------|
| Naive Generation | 30.6% | 27.9% | 10.4% | 22.4% |
| Naive RAG | 49.6% | 29.5% | 12.8% | 32.0% |
| IRCoT | 52.1% | 30.6% | 14.2% | 36.8% |
| Iter-RetGen | 52.2% | 31.9% | 16.1% | 35.2% |
| **ReSearch-7B** | 54.4% | 32.2% | - | - |
| **ReSearch-32B** | **63.6%** | **54.2%** | **33.4%** | **54.4%** |

**关键发现**：
- ReSearch-32B在HotpotQA上达到63.6%，较最强基线提升8.9%
- 在2Wiki上提升22.4%，显示出强大的多跳推理能力
- 7B模型也能超越32B基线方法，证明方法的有效性

#### 3.2.2 泛化能力

ReSearch仅在HotpotQA上训练，但在其他数据集上表现出色：

```
训练集: HotpotQA (单一数据集)
测试集: 2Wiki, MuSiQue, Bamboogle (零样本迁移)
结果: 所有数据集均有显著提升
```

这表明ReSearch学习到了**通用的推理与搜索协作策略**，而非特定数据集的记忆。

### 3.3 消融实验

#### 3.3.1 训练方法对比

| 方法 | HotpotQA | 说明 |
|------|----------|------|
| SFT (监督微调) | 45.2% | 使用人工标注的推理链 |
| RL (强化学习) | 63.6% | 仅使用答案标注 |
| SFT + RL | 64.1% | 先SFT后RL |

**结论**：纯RL训练即可超越SFT，且无需昂贵的推理链标注。

#### 3.3.2 基础模型对比

| 基础模型 | HotpotQA | 参数规模 |
|----------|----------|----------|
| Qwen2.5-7B-Base | 48.3% | 7B |
| Qwen2.5-7B-Instruct | 54.4% | 7B |
| Qwen2.5-32B-Base | 58.7% | 32B |
| Qwen2.5-32B-Instruct | 63.6% | 32B |

**结论**：Instruct版本作为基础模型效果更好，说明预训练的对齐有帮助。

### 3.4 涌现能力分析

ReSearch在训练过程中自然涌现出多种高级推理能力：

#### 3.4.1 反思能力（Reflection）

模型能够识别搜索结果的不足并主动重新搜索：

```

之前搜索"爱因斯坦奖项"返回了很多信息，但没有明确提到诺贝尔奖。
我需要更精确地搜索"爱因斯坦诺贝尔奖"来获取具体年份。
</think>

<search>爱因斯坦 诺贝尔奖 年份</search>
```

#### 3.4.2 自我修正（Self-Correction）

当发现推理错误时，模型能够回溯并修正：

```

等等，我之前的推理有误。诺贝尔奖是1901年开始颁发的，
如果爱因斯坦在1905年提出狭义相对论，他不可能在1901年获奖。
让我重新搜索他的实际获奖年份。
</think>
```

#### 3.4.3 搜索策略优化

模型学会了从粗粒度到细粒度的渐进式搜索：

**案例：爱因斯坦诺贝尔奖问题**

```
Step 1: 搜索 "爱因斯坦 奖项"
→ 获得：诺贝尔奖、马克斯·普朗克奖章等

Step 2: 搜索 "爱因斯坦 诺贝尔奖 年份"
→ 获得：1921年诺贝尔物理学奖

Step 3: 验证信息并给出答案
```

## 四、技术深度解析

### 4.1 为什么ReSearch有效？

#### 4.1.1 从模仿学习到探索学习

传统SFT方法要求模型模仿人工编写的推理链，存在以下问题：
- **标注成本高**：人工编写高质量推理链耗时费力
- **风格单一**：人工推理可能不是最优策略
- **泛化受限**：难以覆盖所有可能的推理路径

ReSearch通过强化学习让模型**自主探索**最优策略：
- 模型尝试不同的搜索时机和查询
- 根据最终答案的正确性获得反馈
- 逐步优化搜索与推理的协作模式

#### 4.1.2 奖励信号稀疏性的处理

多跳问答的奖励信号非常稀疏（仅在最后有答案正确性反馈），ReSearch通过以下方式处理：

1. **GRPO的组内对比**：通过对比同组不同样本的相对表现，获得更细粒度的信号
2. **格式奖励的辅助**：确保模型保持正确的推理结构
3. **多次采样**：每个问题生成多个候选答案，增加学习效率

#### 4.1.3 避免过拟合的设计

```
关键设计：屏蔽检索结果对损失的干扰

传统做法：检索结果参与梯度计算
问题：模型可能过拟合训练时的特定检索结果

ReSearch做法：
- 仅答案部分参与损失计算
- 检索结果作为环境反馈，不直接影响策略梯度
- 迫使模型学习通用的搜索策略，而非记忆特定结果
```

### 4.2 与相关工作对比

#### 4.2.1 与DeepSeek-R1的关系

| 维度 | DeepSeek-R1 | ReSearch |
|------|-------------|----------|
| 核心能力 | 纯文本推理 | 推理+搜索 |
| 推理链 | 纯文本思考 | 文本+搜索+结果 |
| 应用场景 | 数学、代码等封闭域 | 开放域多跳问答 |
| 训练方法 | GRPO + 规则奖励 | GRPO + F1奖励 |

ReSearch可以视为将R1的推理能力扩展到需要外部知识的开放域场景。

#### 4.2.2 与RAG方法的对比

| 方法 | 架构 | 训练方式 | 优势 |
|------|------|----------|------|
| Naive RAG | 先检索后生成 | 无训练 | 简单直接 |
| IRCoT | 交错检索与推理 | 有监督 | 结构化推理 |
| Self-RAG | 自适应检索 | 有监督 | 按需检索 |
| **ReSearch** | 检索嵌入推理链 | 强化学习 | 自主学习、涌现能力 |

### 4.3 局限性与挑战

#### 4.3.1 当前局限

1. **检索系统依赖**：ReSearch依赖外部的BM25等检索系统，检索质量影响最终性能
2. **计算开销**：每轮搜索需要实际调用检索系统，推理成本较高
3. **长文本处理**：多轮搜索后的上下文管理仍需优化
4. **评估瓶颈**：LLM-as-a-Judge的评估成本较高

#### 4.3.2 未来方向

1. **联合优化检索器**：将检索模型也纳入端到端训练
2. **高效推理策略**：学习何时停止搜索，避免过度检索
3. **多模态扩展**：将搜索扩展到图像、视频等多模态内容
4. **工具扩展**：除搜索外，整合计算器、代码执行等更多工具

## 五、个人理解与行业影响

### 5.1 个人理解

#### 5.1.1 范式转变：从"工具调用"到"认知扩展"

ReSearch代表了一种重要的范式转变：

**传统视角**：搜索是模型使用的外部工具
- "我需要一个答案，让我搜索一下"
- 工具与主体分离

**ReSearch视角**：搜索是模型认知能力的自然延伸
- "我在思考这个问题，需要补充一些信息"
- 搜索成为推理过程的一部分

这种转变类似于人类认知：我们不会在思考前就决定"我要查资料"，而是在思考过程中自然地意识到"我需要更多信息"。

#### 5.1.2 强化学习的潜力再发现

ReSearch再次证明了强化学习在大模型后训练中的巨大潜力：

- **超越模仿**：RL可以让模型发现人类未示范过的策略
- **涌现能力**：反思、自我修正等高级能力可以自然涌现
- **标注效率**：仅需答案标注，大幅降低数据成本

这与AlphaGo、AlphaZero的成功逻辑一致：给定正确的奖励信号，模型可以自主发现复杂的策略模式。

#### 5.1.3 对Agent架构的启示

ReSearch为AI Agent的架构设计提供了重要启示：

```
传统Agent架构：
感知 → 规划 → 工具选择 → 执行 → 观察 → ...
（模块化，人工设计交互逻辑）

ReSearch启发的Agent架构：
统一推理链（感知+思考+工具调用+观察融合）
（端到端，自主学习中决策）
```

这种统一架构可能更适合构建真正自主的AI Agent。

### 5.2 行业影响

#### 5.2.1 对企业应用的影响

1. **企业知识库问答**
   - 传统RAG需要大量调优
   - ReSearch可以自主学习最优检索策略
   - 大幅降低部署成本

2. **智能客服**
   - 复杂问题需要多轮信息查询
   - ReSearch能够自主决定何时查询知识库
   - 提升问题解决率和用户满意度

3. **研究辅助**
   - 文献综述需要跨多篇论文检索
   - ReSearch适合处理这种多跳信息整合任务

#### 5.2.2 对学术研究的影响

1. **新的研究范式**
   - 证明RL在开放域推理中的有效性
   - 可能催生更多"推理+X"的融合框架

2. **数据效率**
   - 仅需答案标注的训练方式
   - 可能改变数据集构建的标准做法

3. **可解释性**
   - 显式的思考+搜索链
   - 比端到端黑盒更具可解释性

#### 5.2.3 技术发展趋势

基于ReSearch和其他近期工作，可以预测以下趋势：

**趋势1：推理能力的工具化**
- 推理不再是模型的"固定能力"
- 而是通过RL不断优化的"动态能力"

**趋势2：搜索与推理的深度融合**
- 从"检索-阅读-回答"到"思考-搜索-整合"
- 边界越来越模糊

**趋势3：自主学习成为主流**
- 减少对人工标注的依赖
- 通过环境反馈自主学习

### 5.3 实际应用建议

对于希望应用ReSearch技术的团队，建议：

1. **从简单场景开始**
   - 先在内部知识库问答上验证
   - 逐步扩展到更复杂的场景

2. **重视检索质量**
   - 检索系统的质量是性能上限
   - 考虑结合稠密检索和稀疏检索

3. **持续优化奖励函数**
   - 基础版的F1奖励可能不够精细
   - 可以尝试加入中间步骤奖励

4. **关注成本效益**
   - 多轮搜索会增加推理成本
   - 需要权衡性能与成本的平衡

## 六、总结与展望

### 6.1 核心贡献总结

ReSearch的核心贡献可以概括为：

1. **提出新框架**：首次将搜索操作深度嵌入推理链，实现端到端强化学习
2. **无需中间标注**：仅需答案标注即可训练，大幅降低数据成本
3. **涌现高级能力**：反思、自我修正等能力通过RL自然涌现
4. **强大泛化性**：单一数据集训练，多数据集表现出色

### 6.2 未来展望

ReSearch为大型语言模型的推理能力发展指明了重要方向：

**短期（1年内）**：
- 更多模型规模（70B+）的验证
- 与其他工具（代码执行、数据库查询）的整合
- 更高效的推理策略（减少搜索轮次）

**中期（1-3年）**：
- 端到端的检索器联合训练
- 多模态搜索（图像、视频、音频）
- 实时信息源（搜索引擎API）的整合

**长期（3-5年）**：
- 完全自主的AI Agent
- 主动学习与持续优化
- 跨领域知识整合与推理

### 6.3 一句话评价

> ReSearch证明了：通过正确的强化学习框架，大模型可以自主学会如何"边思考边搜索"，无需人工示范即可掌握复杂的多跳推理能力，这为构建真正自主的AI Agent迈出了关键一步。
