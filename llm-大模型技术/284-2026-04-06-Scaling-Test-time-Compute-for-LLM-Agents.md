# LLM Agent测试时计算扩展的系统探索：从并行采样到智能反思

> **论文标题**: Scaling Test-time Compute for LLM Agents  
> **作者**: King Zhu, Hanhao Li, Siwei Wu, Tianshun Xing, Dehua Ma, Xiangru Tang, Minghao Liu, Jian Yang, Jiaheng Liu, Yuchen Eleanor Jiang, Changwang Zhang, Chenghua Lin, Jun Wang, Ge Zhang, Wangchunshu Zhou  
> **发表时间**: 2025年6月  
> **arXiv**: https://arxiv.org/abs/2506.12928  

---

## 一、研究背景与核心问题

### 1.1 测试时计算扩展的崛起

大语言模型（LLMs）的推理能力在过去两年经历了质的飞跃。从2024年OpenAI发布O1模型开始，"长思考"（long-thinking）模型展示了通过增加推理时间可以显著提升复杂任务解决能力的可能性。DeepSeek-R1、Qwen3等后续模型进一步验证了**测试时计算扩展**（Test-Time Scaling, TTS）这一范式的有效性。

传统上，提升模型性能主要依靠两个维度：**扩大模型规模**（更多参数）和**增加训练数据**。然而，这两个维度都面临边际收益递减和成本激增的挑战。测试时计算扩展开辟了第三条路径：**在推理阶段投入更多计算资源**，通过多次采样、自我修正、树搜索等方式，让模型"思考更久、探索更深"。

### 1.2 智能体（Agent）的特殊挑战

尽管测试时计算扩展在纯文本推理任务（如数学解题、代码生成）上取得了显著成功，但将其应用于**语言智能体**（Language Agents）面临独特的挑战：

**第一，步骤复杂性差异**。传统LLM解决特定问题通常是端到端的单轮或少数几轮交互。而智能体需要将复杂问题分解为多个步骤，依次调用不同工具和模型，形成长序列的决策链。这意味着错误会在多步骤中累积，传统的TTS方法（如Best-of-N）如果在每一步都随机生成N个响应，可能导致最终结果严重偏离正确路径。

**第二，工具调用的特殊性**。智能体框架涉及代码执行、API调用、文件操作等多种工具使用场景。这些操作具有状态依赖性——前面的步骤会影响后续步骤的执行环境。这使得简单的并行采样策略难以直接迁移。

**第三，验证的困难性**。在数学推理中，答案往往可以通过程序验证或标准答案比对来评判。但在智能体任务中，涉及网页浏览、多模态文件处理、开放域问答等，验证模型的行为质量变得更加复杂。

### 1.3 研究动机与目标

基于上述背景，本文作者提出了核心研究问题：**如何将测试时计算扩展方法有效地应用于语言智能体框架？**

具体来说，论文系统探索了四个关键维度：
1. **并行采样算法**：Best-of-N、Beam Search、DVTS等方法在智能体框架中的适配与效果
2. **序列修订策略**：反思（Reflection）机制的最佳应用时机和频率
3. **验证与结果合并**：如何设计有效的过程奖励模型（PRM）和结果选择策略
4. **Rollout多样化**：通过多模型协作增强探索空间的策略

## 二、技术方法详解

### 2.1 ATTS框架总体架构

论文提出的**Agentic Test-Time Scaling (ATTS)**框架包含四个核心组件，形成完整的测试时计算扩展流水线：

```
用户查询 → 并行采样/序列修订 → 验证器评估 → 结果合并 → 最终答案
                ↑                      ↓
            Rollout多样化 ←── 过程奖励反馈
```

框架基于SmoLAgents构建，移除了ToolAgent的嵌套结构，直接使用CodeAgent进行工具调用，以GPT-4.1作为主要基线模型。

### 2.2 并行采样算法对比

论文实现了四种主流的并行采样算法，并进行了系统性对比：

#### 2.2.1 Best-of-N (BoN)

BoN是最简单的并行采样策略。对于给定的问题Q，从LLM独立采样N个响应：

```
{R₁, R₂, ..., Rₙ} = BoN(Q)
```

然后通过验证模型选择最优答案。BoN的有效性高度依赖于两个因素：**奖励模型的质量**和**采样候选的多样性**。

在智能体框架中，BoN在整个任务级别执行——即生成N条完整的任务执行轨迹，然后选择最优的一条。

#### 2.2.2 Step-wise Best-of-N (BoN-wise)

与BoN在任务级别采样不同，BoN-wise在**每个决策步骤**都进行N次采样：

给定前t-1步的思考{T₁, T₂, ..., Tₜ₋₁}，在第t步时：

```
{R₁, R₂, ..., Rₙ} = BoN-Wise(Q, T₁, T₂, ..., Tₜ₋₁)
```

然后选择最优响应作为第t步的思考Tₜ。

BoN-wise允许在每个决策节点进行最大规模的探索，特别适合需要精细步骤选择的复杂任务。

#### 2.2.3 Beam Search

Beam Search维护固定大小的候选集（beam size K）。在每个步骤：

1. 对每个当前候选节点生成N个后续响应
2. 使用验证模型评估所有K×N个候选
3. 保留得分最高的K个候选进入下一步

形式化表示：

```
{R₁, R₂, ..., Rₖ} = BeamSearch(Q, LNᵢ)
```

其中LNᵢ表示当前第i个叶子节点。Beam Search通过早期剪枝减少搜索空间，依赖于累积对数概率或奖励分数进行候选筛选。

#### 2.2.4 DVTS (Diverse Verifier Tree Search)

DVTS将搜索空间分解为K个子树，每个子树独立执行Beam Search：

```
{R₁, R₂, ..., Rₖ} = DVTS(Q, SubTreeᵢ)
```

通过并行探索多个子树，DVTS实现了比单一深度Beam Search更丰富的搜索行为。在相同计算预算下，DVTS能找到更高质量的解决方案。

### 2.3 序列修订策略：反思的时机与频率

反思（Reflection）是测试时计算扩展中的重要机制，允许智能体从错误中学习并自我修正。论文提出了**阈值驱动的反思策略**：

#### 2.3.1 反思模型设计

给定前t-1步的思考历史{T₁, T₂, ..., Tₜ₋₁}，反思模型RefM生成状态总结：

```
Sumₜ = RefM(T₁, T₂, ..., Tₜ₋₁)
```

这个总结包含对当前执行状态的评估、已获取的信息、以及可能的改进方向。

#### 2.3.2 阈值触发机制

关键创新在于**何时触发反思**。论文使用验证模型对每个步骤进行客观评分（0-10分），并设置不同的阈值：

- **频繁反思**：阈值<8，只要分数低于8就触发
- **适度反思**：阈值<5，分数低于5才触发  
- **选择性反思**：阈值<2，仅在严重错误时触发

只有当当前步骤得分低于阈值时，才会将反思总结Sumₜ注入LLM的上下文，指导后续生成。

这一设计的核心洞察是：**知道何时反思比每步都反思更重要**。频繁的反思会打断模型的推理流，反而降低性能；而选择性的、针对性的反思能够最大化收益。

### 2.4 验证器与结果合并方法

#### 2.4.1 过程奖励模型（PRM）

论文设计了两种PRM评估方式：

**Scoring PRM**：对每个中间步骤进行独立评分

对于N个候选步骤{T₁, T₂, ..., Tₙ}，使用奖励模型RM计算每个候选的分数：

```
Sᵢ = RM(T_current_i)
```

**List-wise PRM**：通过直接比较选择最优轨迹

将所有N个候选同时提供给LLM，让其选择最优的一个：

```
Sᵢ = RM(T_current_i)  // 通过比较得出
```

List-wise方法的优势在于提供了相对比较的基准，避免了绝对评分的主观性。

#### 2.4.2 结果合并策略

论文比较了三种主流的结果合并方法：

1. **投票（Voting）**：选择出现频率最高的答案
2. **评分（Scoring）**：使用PRM直接评分，选择最高分
3. **列表式（List-wise）**：模型直接从候选中选择最优答案

实验结果显示，List-wise方法在各类并行采样算法中都表现最佳，因为它既利用了相对比较的准确性，又能从多样化候选中识别潜在正确答案。

### 2.5 Rollout多样化策略

#### 2.5.1 超参数控制

通过调整temperature、top_p等采样参数，可以控制生成多样性。但论文发现，仅靠超参数调整难以充分扩展探索空间。

#### 2.5.2 多模型协作采样

更强大的策略是使用**不同的LLM作为rollout模型**。不同模型具有不同的能力画像：
- 某些模型擅长代码生成
- 某些模型在工具使用上表现优异
- 某些模型在推理规划方面更强

论文设计了多智能体协作框架，使用GPT-4.1、Claude-3-5、Claude-3-7、Gemini-2.5-Pro等不同模型的组合进行并行采样。这种方法显著增加了rollout的多样性，提升了找到正确答案的概率。

## 三、实验设计与结果分析

### 3.1 实验设置

**基准测试**：GAIA验证集（165个样本，分三个难度等级）
- Level 1：基础网页搜索和文件处理
- Level 2：中等复杂度，需要多步推理
- Level 3：高复杂度，需要深度研究和复杂工具链

**基线模型**：GPT-4.1（主要实验）、Claude-3-5、Claude-3-7、Gemini-2.5-Pro（多模型实验）

**评估指标**：任务成功率（按难度分层）

### 3.2 主要实验结果

#### 3.2.1 并行采样算法对比

| 方法 | 平均得分 | Level 1 | Level 2 | Level 3 |
|------|----------|---------|---------|---------|
| Baseline | 55.76 | 66.04 | 58.14 | 26.92 |
| BoN | **63.03** | **77.36** | **63.95** | 30.77 |
| BoN-wise | 58.79 | 69.23 | 58.62 | **38.46** |
| Beam Search | 56.97 | 69.81 | 55.81 | 34.62 |
| DVTS | 55.76 | 58.49 | 62.79 | 26.92 |

**关键发现**：

1. **BoN表现最优**，相比基线提升8分，在Level 1和Level 2达到SOTA水平。BoN通过端到端的多次尝试，特别适合需要工具调用的中等难度任务。

2. **BoN-wise在最难任务上表现最好**，Level 3得分38.46，超越BoN。这表明增加步骤级探索对复杂任务更有帮助。

3. **Beam Search和DVTS提升有限**，原因是这些算法虽然增加了探索空间，但严重依赖验证模型提供的信号质量。在智能体任务中，中间步骤的验证信号往往不够稳定，导致搜索方向偏离。

#### 3.2.2 反思策略的效果

| 策略 | 平均得分 | Level 1 | Level 2 | Level 3 |
|------|----------|---------|---------|---------|
| Baseline | 55.76 | 66.04 | 58.14 | 26.92 |
| 每步反思 | 55.15 | 71.70 | 51.16 | 34.62 |
| 阈值<8 | 53.33 | 66.04 | 53.49 | 26.92 |
| 阈值<5 | 52.12 | 69.81 | 50.00 | 23.08 |
| **阈值<2** | **56.36** | **71.70** | **55.81** | 26.92 |

**关键洞察**：

- **每步反思反而降低整体性能**（55.15 vs 55.76），说明过度反思会干扰推理流
- **选择性反思（阈值<2）表现最佳**，平均56.36，超越基线
- **频繁反思（阈值<8）损害Level 2性能**，因为中等复杂度任务最容易被过度反思打断

这一结果强有力地证明了**反思时机的重要性**：只在真正需要时反思，才能最大化收益。

#### 3.2.3 验证与合并方法对比

**不同结果合并方法的效果**（基于BoN）：

| 合并方法 | BoN得分 | Beam Search得分 |
|----------|---------|-----------------|
| 投票 | 56.80 | 54.55 |
| 评分 | 59.39 | 53.94 |
| **列表式** | **63.03** | **56.97** |

**不同验证方法的效果**：

| 搜索方法 | 评分PRM | 列表式PRM |
|----------|---------|-----------|
| BoN-wise | 56.36 | **58.79** |
| Beam Search | 53.94 | **56.97** |
| Tree Search | 50.91 | **55.76** |

**核心结论**：无论是结果合并还是过程验证，**列表式方法都显著优于其他方法**（平均提升3分）。列表式验证通过相对比较而非绝对评分，提供了更可靠的评估基准。

#### 3.2.4 Rollout多样化的效果

**采样宽度的影响**：
- 采样宽度从1增加到4，BoN性能持续提升（符合LLM领域的TTS现象）
- BoN-wise在Level 3任务上从宽度增加中获益更大

**多模型协作的效果**（Pass@K指标）：

| Model组合 | Pass@1 | Pass@2 | Pass@4 |
|-----------|--------|--------|--------|
| GPT-4.1 only | 55.76 | 60.49 | 69.14 |
| 4模型混合 | - | 64.24 | **74.55** |

使用GPT-4.1、Claude-3-5、Claude-3-7、Gemini-2.5-Pro四模型混合，Pass@4达到74.55，**超越开源SOTA水平**。

这一结果验证了**多样化rollout的价值**：不同模型的能力互补增加了找到正确答案的概率。

### 3.3 与SOTA方法的对比

论文将ATTS与当前主流的智能体框架进行了对比：

| 框架 | 模型 | 平均得分 |
|------|------|----------|
| TapeAgents | Claude-3-7等 | 55.76 |
| AutoAgent | Claude-3-5等 | 55.15 |
| Open Deep Research | OpenAI o1 | 55.15 |
| Magnetic-1 | OpenAI o1等 | 46.06 |
| Smolagents | OpenAI o1等 | 53.33 |
| **ATTS (BoN)** | **GPT-4.1** | **63.03** |

ATTS（使用BoN）显著超越所有对比方法，证明了测试时计算扩展在智能体领域的有效性。

## 四、个人理解与行业影响分析

### 4.1 理论贡献深度解析

#### 4.1.1 测试时计算扩展的普适性验证

本文最重要的理论贡献是**验证了测试时计算扩展在智能体领域的普适性**。此前，TTS的有效性主要在数学推理、代码生成等封闭域任务上得到证明。本文通过系统实验表明，在涉及工具调用、网页浏览、多模态处理的开放域智能体任务中，TTS同样有效。

这一发现具有重要意义：它意味着**推理时的计算投资可以作为一种通用能力增强手段**，不局限于特定任务类型。

#### 4.1.2 "反思时机"的范式启示

本文关于反思策略的发现——**知道何时反思比每步都反思更重要**——挑战了直觉上的"越多反思越好"假设。

这一结果可以从认知科学角度理解：
- **元认知监控**：人类在解决问题时，不会每一步都进行自我评估，而是在遇到障碍或不确定性时才停下来反思
- **认知负荷理论**：频繁的反思增加了工作记忆负担，可能干扰主任务的执行
- **流状态**：过度中断会破坏问题解决的"流状态"

这一发现对智能体设计具有直接指导意义：**反思机制应该是有条件触发的，而非默认开启**。

#### 4.1.3 列表式验证的认知优势

列表式验证方法的优越性可以从认知心理学角度解释：
- **相对比较优于绝对判断**：人类在评估多个选项时，相对比较往往比独立评分更准确
- **消除评分标准漂移**：独立评分容易受到评分者标准变化的影响，而相对比较在同一上下文中进行
- **利用相对排序的一致性**：即使绝对评分不准确，相对排序往往更稳定

这一发现对奖励模型设计具有深远影响：**过程奖励模型应该采用比较式而非评分式架构**。

### 4.2 实践应用价值

#### 4.2.1 智能体框架设计指南

本文提供了清晰的设计原则：

1. **并行采样策略选择**：
   - 中等复杂度任务（Level 1-2）：优先使用BoN
   - 高复杂度任务（Level 3）：考虑BoN-wise
   - Beam Search和DVTS需要更强的验证信号支持

2. **反思机制设计**：
   - 使用验证模型评估每步质量
   - 设置合理的阈值（建议<2分才触发）
   - 避免每步反思的过度开销

3. **验证器实现**：
   - 优先采用列表式比较
   - 评估指标应包括：任务进展、错误严重性、循环检测、反思利用率

4. **多样化策略**：
   - 组合不同能力画像的模型
   - 4个模型以上的组合收益递减

#### 4.2.2 成本效益分析

测试时计算扩展的成本模型：
- **BoN策略**：N倍推理成本，线性提升
- **BoN-wise策略**：N×T倍推理成本（T为平均步骤数），但复杂任务提升更明显
- **多模型策略**：固定成本增加（取决于模型组合），但pass@k显著提升

在实际部署中，应该根据任务难度动态选择策略：
- 简单任务：基线模型直接回答
- 中等任务：BoN（N=4）
- 复杂任务：BoN-wise + 选择性反思

#### 4.2.3 与训练时优化的权衡

本文结果与Snell等人的发现一致：**最优的测试时计算扩展可能比扩大模型参数更有效**。

对于一个32B参数模型：
- 训练更大模型（如70B）：需要数亿美元训练成本
- 使用测试时计算扩展（BoN-4）：推理成本增加4倍，但可能达到相似或更好的性能

这为资源受限的场景提供了实用路径：**优先投资推理基础设施，而非盲目追求更大模型**。

### 4.3 行业影响与前景展望

#### 4.3.1 对智能体平台的冲击

本文的发现将深刻影响智能体平台的设计：

1. **LangChain、SmoLAgents等框架**需要集成TTS能力
2. **云服务提供商**需要优化批量推理和并行采样基础设施
3. **企业级应用**可以通过配置TTS策略来平衡成本与性能

#### 4.3.2 多模态智能体的启示

虽然本文主要在文本智能体上验证，但其方法论可以直接扩展到多模态智能体：
- 视觉-语言模型（VLM）的工具使用
- 具身智能体的决策规划
- 自动驾驶的复杂场景推理

#### 4.3.3 研究方向的引领

本文开启了多个值得深入的研究方向：

1. **自适应TTS**：根据任务难度动态调整采样宽度和反思阈值
2. **学习式验证器**：训练专门的过程奖励模型替代通用LLM评估
3. **层次化TTS**：在任务级、步骤级、子目标级分别应用TTS
4. **多智能体协作TTS**：探索智能体间的知识共享和协同搜索

#### 4.3.4 潜在局限与未来工作

本文的局限包括：
- 主要在GAIA基准上验证，需要更多样化的任务验证
- 验证模型本身的质量成为瓶颈
- 多模型协作的成本仍然较高

未来工作可以探索：
- 更高效的验证信号获取方式
- 模型蒸馏将TTS能力压缩到单次推理
- 在线学习动态优化TTS策略

## 五、总结与关键洞察

### 5.1 核心贡献回顾

本文作为**首个系统探索测试时计算扩展在LLM智能体上应用的研究**，做出了以下关键贡献：

1. **全面的方法论框架**：提出了ATTS框架，涵盖并行采样、序列修订、验证合并、多样化rollout四个维度

2. **重要的实验发现**：
   - BoN在智能体任务上表现最优
   - 选择性反思（阈值<2）比每步反思更有效
   - 列表式验证方法显著优于评分和投票
   - 多模型协作可以显著提升pass@k

3. **实践指导价值**：为智能体框架设计提供了清晰的策略选择指南

### 5.2 关键洞察提炼

**洞察一：简单策略往往最有效**

在并行采样算法中，最简单的BoN超越了更复杂的Beam Search和DVTS。这表明在智能体任务中，**端到端的多次尝试比精细的中间步骤控制更可靠**。

**洞察二：时机比频率更重要**

反思机制的设计应该关注"何时反思"而非"是否反思"。**阈值驱动的选择性反思**能够最大化收益同时最小化干扰。

**洞察三：相对比较优于绝对评估**

无论是过程验证还是结果合并，列表式方法（相对比较）都优于评分式方法（绝对评估）。这一发现具有普适性，可以指导各类评估系统的设计。

**洞察四：多样性带来鲁棒性**

多模型协作的显著效果表明，**能力的多样性比单一模型的强度更重要**。这为智能体系统的架构设计提供了新思路。

### 5.3 对AI发展的意义

测试时计算扩展代表了AI系统设计的范式转变：

**从"更大模型"到"更聪明的推理"**：不再单纯追求参数规模，而是通过优化推理过程来提升能力。

**从"单次生成"到"迭代优化"**：引入类似人类思考的"尝试-反思-修正"循环。

**从"单一模型"到"模型协作"**：利用不同模型的能力互补实现系统级增强。

本文的研究为这一范式转变在智能体领域的落地提供了重要实证支撑，预示着一个**"推理优先"**的新时代正在到来。
