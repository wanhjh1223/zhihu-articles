## 一、研究背景与核心问题

### 1.1 检索增强生成的发展现状

检索增强生成（Retrieval-Augmented Generation, RAG）作为大型语言模型（LLMs）获取和利用外部知识的核心技术，已经在各种领域应用中发挥着至关重要的作用。RAG技术通过将外部知识库的检索能力与LLM的生成能力相结合，有效解决了大模型知识时效性不足、容易产生幻觉（hallucination）等固有问题。

近年来，随着大模型能力的不断提升，RAG技术也经历了从单轮检索到多轮迭代检索的演进。传统的单轮RAG在面对简单事实性查询时表现良好，但在处理复杂的多跳推理（multi-hop reasoning）问题时却显得力不从心。多跳问题要求模型能够跨多个文档进行推理，逐步收集证据，最终得出答案。

### 1.2 现有多轮RAG方法的核心挑战

为了增强复杂任务的搜索能力，现有的大多数研究工作通过端到端训练将多轮迭代检索与推理过程相结合。这些方法虽然在问题解决性能上取得了显著提升，但仍面临着两个根本性挑战：

**挑战一：模糊的检索执行路径**

在端到端的强化学习训练过程中，模型需要同时学习"如何规划"（planning）和"如何执行"（execution）。这种耦合的训练方式导致模型难以形成清晰的检索执行路径，往往在检索过程中出现方向性偏差，无法准确判断下一步应该检索什么内容。

**挑战二：稀疏奖励信号问题**

端到端强化学习训练中，模型通常只能在任务完成时获得奖励信号，而在中间检索步骤中缺乏细粒度的指导。这种稀疏的奖励机制使得模型难以学到有效的中间策略，导致检索结果不准确，最终影响整体性能。

### 1.3 研究动机与创新思路

针对上述挑战，APEX-Searcher提出了一种全新的智能体规划与执行框架（Agentic Planning and Execution Framework）。该框架的核心创新在于将检索过程解耦为两个阶段：**战略规划阶段**和**子任务执行阶段**。

这种解耦设计的灵感来源于人类解决复杂问题的方式：当人类面对复杂任务时，通常会先制定一个宏观的执行计划（将大问题分解为若干子任务），然后按照计划逐步执行每个子任务。APEX-Searcher正是模仿了这种"先规划后执行"的问题解决范式。

## 二、APEX-Searcher技术框架详解

### 2.1 整体架构概述

APEX-Searcher采用双阶段智能体框架，将检索过程明确分解为两个互补的阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│                    APEX-Searcher Framework                      │
├─────────────────────────┬───────────────────────────────────────┤
│   Stage 1: Planning     │      Stage 2: Execution               │
│   (战略规划阶段)         │      (子任务执行阶段)                  │
├─────────────────────────┼───────────────────────────────────────┤
│ • RL训练               │ • 监督微调(SFT)                        │
│ • 分解特定奖励          │ • 高质量多跳轨迹                       │
│ • 战略优化             │ • 迭代子任务执行                       │
│ • 任务分解             │ • 鲁棒的检索能力                       │
└─────────────────────────┴───────────────────────────────────────┘
```

### 2.2 第一阶段：战略规划（Planning）

#### 2.2.1 任务分解策略

在规划阶段，APEX-Searcher首先将复杂的原始问题分解为一系列有序的、可管理的子任务。每个子任务都有明确的目标和预期输出，形成一个结构化的任务依赖图。

例如，对于问题"苹果公司2024年第二季度营收增长的主要驱动因素是什么？"，系统可能将其分解为：
- 子任务1：检索苹果公司2024年Q2财报
- 子任务2：提取营收数据并与去年同期对比
- 子任务3：识别各业务板块的收入贡献
- 子任务4：分析主要增长驱动因素

#### 2.2.2 强化学习与分解特定奖励

APEX-Searcher使用强化学习（RL）来优化战略规划过程。与传统RL方法不同，该框架引入了**分解特定奖励（Decomposition-Specific Rewards）**机制：

**奖励设计原则**：
- **子任务完成奖励**：每个子任务成功完成时给予正奖励
- **任务连贯性奖励**：子任务之间的逻辑连贯性得到奖励
- **信息增益奖励**：每次检索带来的有效信息增益被量化并奖励
- **冗余惩罚**：重复检索或无效检索受到惩罚

这种细粒度的奖励设计解决了传统端到端RL中的稀疏奖励问题，使模型能够在每个子任务步骤中获得学习信号。

#### 2.2.3 策略优化目标

规划阶段的策略优化目标是学习一个最优的子任务分解策略π*，使得：

```
π* = argmax E[R_total | π]

其中 R_total = Σ(γ^t * r_t)
- γ: 折扣因子
- r_t: 第t步的分解特定奖励
```

通过这种方式，模型学会如何将复杂查询分解为最优的子任务序列。

### 2.3 第二阶段：子任务执行（Execution）

#### 2.3.1 监督微调策略

在执行阶段，APEX-Searcher采用监督微调（SFT）方法来训练模型具备鲁棒的迭代子任务执行能力。与规划阶段不同，执行阶段关注的是"如何高质量地完成每个子任务"。

#### 2.3.2 高质量多跳轨迹构建

为了训练执行模型，APEX-Searcher构建了高质量的多跳推理轨迹数据集。这些轨迹包含：

- **完整的检索路径**：从初始查询到最终答案的完整检索链路
- **中间推理步骤**：每个检索步骤的思考和决策过程
- **验证机制**：关键信息的多源交叉验证记录
- **错误恢复路径**：遇到无效检索时的修正策略

#### 2.3.3 迭代检索与推理

执行阶段的核心能力是实现迭代检索与推理的深度融合：

```python
# 伪代码示意
def iterative_execution(subtask, retriever, llm):
    context = []
    for step in range(max_steps):
        # 基于当前上下文生成检索查询
        query = llm.generate_query(subtask, context)
        
        # 执行检索
        retrieved_docs = retriever.search(query)
        
        # 评估检索结果的相关性
        relevance = llm.assess_relevance(retrieved_docs, subtask)
        
        # 如果获得足够相关信息，生成答案
        if relevance > threshold:
            answer = llm.generate_answer(context + retrieved_docs)
            return answer
        
        # 否则，更新上下文并继续迭代
        context.extend(retrieved_docs)
        
        # 生成下一步的细化查询
        subtask = llm.refine_subtask(subtask, retrieved_docs)
    
    return llm.generate_answer(context)  # 返回最佳努力答案
```

### 2.4 两阶段协同机制

APEX-Searcher的两个阶段并非独立运作，而是通过精心设计的接口进行协同：

1. **规划到执行的接口**：规划阶段输出的子任务序列作为执行阶段的输入
2. **执行反馈机制**：执行阶段的中间结果可以触发规划的动态调整
3. **端到端优化**：虽然训练是分离的，但推理时可以形成端到端的优化闭环

## 三、实验设计与结果分析

### 3.1 实验设置

#### 3.1.1 评测基准

APEX-Searcher在多个权威的RAG和任务规划基准上进行了评测：

**多跳RAG基准**：
- HotpotQA：需要跨多个文档进行多跳推理的问答数据集
- 2WikiMultiHopQA：基于Wikipedia的多跳问答数据集
- MuSiQue：多步骤问题解答数据集

**任务规划基准**：
- ToolBench：工具使用任务规划数据集
- WebShop：电商购物任务规划
- ALFWorld：室内导航和交互任务

#### 3.1.2 对比方法

APEX-Searcher与以下方法进行了对比：
- **Baseline RAG**：标准的单轮RAG方法
- **Iterative RAG**：简单的迭代检索方法
- **Self-RAG**：自适应检索的RAG方法
- **ReAct**：推理与行动结合的LLM智能体
- **DSPy**：声明式语言模型编程框架

### 3.2 主要实验结果

#### 3.2.1 多跳RAG性能

在多跳问答任务上，APEX-Searcher取得了显著的性能提升：

| 方法 | HotpotQA | 2WikiMultiHopQA | MuSiQue |
|------|----------|-----------------|---------|
| Baseline RAG | 42.3 | 38.7 | 28.5 |
| Iterative RAG | 51.2 | 45.3 | 35.8 |
| Self-RAG | 56.8 | 49.6 | 39.2 |
| ReAct | 58.4 | 51.2 | 41.5 |
| DSPy | 60.1 | 53.8 | 43.7 |
| **APEX-Searcher** | **67.5** | **61.3** | **49.2** |

从结果可以看出，APEX-Searcher在所有三个基准上都取得了最佳性能，相比最强的基线方法DSPy，在HotpotQA上提升了7.4个百分点，在2WikiMultiHopQA上提升了7.5个百分点，在MuSiQue上提升了5.5个百分点。

#### 3.2.2 任务规划性能

在任务规划任务上，APEX-Searcher同样表现出色：

| 方法 | ToolBench | WebShop | ALFWorld |
|------|-----------|---------|----------|
| ReAct | 62.5 | 58.3 | 71.2 |
| Plan-and-Solve | 65.8 | 61.5 | 73.8 |
| Tree of Thoughts | 68.2 | 64.7 | 76.5 |
| **APEX-Searcher** | **74.6** | **70.3** | **82.1** |

APEX-Searcher在任务规划基准上相比最佳基线Tree of Thoughts有6.4%到10.6%的性能提升。

#### 3.2.3 消融实验

为了验证两阶段解耦设计的有效性，论文进行了详细的消融实验：

**实验设置**：
- 完整模型：APEX-Searcher（两阶段解耦）
- 变体1：端到端联合训练（不解耦）
- 变体2：仅规划阶段
- 变体3：仅执行阶段

**实验结果**（HotpotQA）：
- 完整模型：67.5
- 端到端联合训练：58.9（-8.6）
- 仅规划阶段：52.3（-15.2）
- 仅执行阶段：61.4（-6.1）

消融实验结果表明：
1. **两阶段解耦显著优于端到端联合训练**，验证了架构设计的有效性
2. **两个阶段相互补充**，单独使用任一阶段都会导致性能下降
3. 规划阶段的缺失对性能影响更大，说明正确的任务分解是成功的基础

### 3.3 效率分析

除了准确性提升外，APEX-Searcher在效率方面也表现出色：

#### 3.3.1 检索轮次分析

| 方法 | 平均检索轮次 | 答案准确率 |

| Iterative RAG | 4.8 | 51.2 |
| Self-RAG | 3.5 | 56.8 |

APEX-Searcher在达到更高准确率的同时，所需的平均检索轮次最少，体现了其高效的检索策略。

#### 3.3.2 计算成本分析

在计算资源消耗方面，APEX-Searcher展现出良好的成本效益比：

- 相比ReAct，APEX-Searcher减少了35%的API调用次数
- 相比端到端RL方法，训练时间缩短了40%

## 四、技术贡献与创新点分析

### 4.1 核心技术创新

#### 4.1.1 规划与执行的显式解耦

APEX-Searcher最重要的技术贡献是将传统耦合的检索-推理过程显式解耦为规划和执行两个阶段。这种解耦带来了多重优势：

**优势一：降低学习难度**
- 规划阶段只需关注"分解策略"
- 执行阶段只需关注"子任务完成"
- 每个阶段的学习目标更加明确，降低了优化难度

**优势二：提升可解释性**
- 生成的子任务序列提供了清晰的推理路径
- 便于调试和错误分析
- 支持人工干预和指导

**优势三：增强模块化**
- 两个阶段可以独立优化和升级
- 便于集成新的规划策略或执行能力
- 支持领域特定的定制化

#### 4.1.2 分解特定奖励机制

针对RL训练中的稀疏奖励问题，APEX-Searcher提出了分解特定奖励机制。这种奖励设计的创新之处在于：

- **多维度奖励信号**：不仅关注最终结果，还关注中间过程的质量
- **任务自适应**：奖励函数可以根据不同任务类型动态调整
- **细粒度反馈**：每个子任务都有明确的奖励信号，加速了学习过程

#### 4.1.3 高质量轨迹构建方法

在执行阶段的训练中，高质量多跳轨迹的构建是关键。APEX-Searcher采用了：

- **专家演示学习**：结合人工标注和自动生成的专家轨迹
- **轨迹过滤机制**：基于最终答案正确性筛选训练样本
- **多样性增强**：通过查询改写和路径采样增加轨迹多样性

### 4.2 与相关工作的对比

#### 4.2.1 与ReAct的对比

ReAct将推理（Reasoning）和行动（Acting）交织进行，但并未显式区分规划和执行：

| 维度 | ReAct | APEX-Searcher |
|------|-------|---------------|
| 架构 | 推理-行动交织 | 规划-执行解耦 |
| 训练 | 提示工程为主 | RL + SFT |
| 可解释性 | 中等 | 高 |
| 可扩展性 | 有限 | 强 |

#### 4.2.2 与Self-RAG的对比

Self-RAG关注自适应检索决策，但没有显式的任务分解机制：

- Self-RAG：每一步决定"是否需要检索"
- APEX-Searcher：先规划"需要检索什么"，再执行检索

#### 4.2.3 与Tree of Thoughts的对比

Tree of Thoughts使用树形搜索探索多个推理路径：

- Tree of Thoughts：探索多个可能的推理路径，计算成本较高
- APEX-Searcher：通过规划确定最优路径，更加高效

- 推动RAG从简单的检索-生成向复杂的规划-执行演进

- **实时规划优化**：支持动态的任务分解和调整

## 六、局限性与未来展望

### 6.1 当前局限性

尽管APEX-Searcher取得了显著进展，但仍存在一些局限性：

#### 6.1.1 规划阶段的计算开销

规划阶段的强化学习训练需要大量计算资源，特别是在任务分解空间较大的情况下。这可能限制其在资源受限环境中的应用。

#### 6.1.2 对领域知识的依赖

当前的任务分解策略在一定程度上依赖于通用知识，对于高度专业化的领域，可能需要额外的领域适配训练。

#### 6.1.3 错误累积风险

在多阶段的处理流程中，早期阶段的错误可能传播到后续阶段。虽然APEX-Searcher设计了一定的纠错机制，但在极端情况下仍可能出现错误累积。

### 6.2 未来研究方向

基于APEX-Searcher的框架，未来研究可以在以下方向深入探索：

#### 6.2.1 自适应规划深度

研究如何根据问题复杂度动态调整规划的粒度：
- 简单问题：浅层规划，快速执行
- 复杂问题：深层规划，细致分解
- 实现规划深度的自适应调节

#### 6.2.2 多智能体协作RAG

将APEX-Searcher扩展到多智能体场景：
- 不同智能体负责不同子任务的规划和执行
- 智能体间的协作与竞争机制
- 分布式RAG架构设计
