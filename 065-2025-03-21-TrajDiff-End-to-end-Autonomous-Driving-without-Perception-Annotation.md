## 一、引言：自动驾驶的数据困境

自动驾驶技术的每一次重大突破，往往都伴随着数据成本的急剧上升。从早期的模块化方案到近年来的端到端方案，虽然架构在不断简化，但对标注数据的依赖却愈发严重。

一个鲜为人知但极具冲击力的数据是：**标注1000个驾驶场景的感知数据需要超过7000小时的人工劳动**。这意味着什么？假设一个标注团队每天工作8小时，需要整整875天——超过两年时间——才能完成这1000个场景的标注。这种高昂的数据成本已经成为制约自动驾驶技术规模化发展的核心瓶颈。

传统的端到端自动驾驶系统虽然在架构上实现了从传感器输入到驾驶决策的直接映射，但其训练过程仍然严重依赖感知标注。无论是障碍物检测、车道线识别，还是高精地图构建，都需要大量的人工标注来监督学习。这种"换汤不换药"的做法，本质上并没有解决数据成本的问题。

正是在这样的背景下，**TrajDiff**应运而生。这篇来自澳门大学、中国科学技术大学等机构的最新研究，提出了一种**完全无需感知标注**的端到端自动驾驶框架，通过在NAVSIM基准上达到**87.5 PDMS**的SOTA性能（数据扩展后进一步提升至**88.5 PDMS**），向业界证明了：没有感知标注，我们一样可以训练出顶尖的自动驾驶系统。

## 二、研究背景与动机

### 2.1 端到端自动驾驶的发展脉络

端到端自动驾驶的理念可以追溯到1989年ALVINN项目的开创性工作，但直到近年来深度学习技术的成熟，这一方向才真正进入快速发展期。从架构演进的角度看，端到端自动驾驶经历了以下几个关键阶段：

**阶段一：纯模仿学习时代**
早期的端到端方法，如NVIDIA的PilotNet，直接通过卷积神经网络将前视摄像头图像映射到方向盘转角。这种方法虽然简单直接，但在复杂场景下的泛化能力有限，经常出现"灾难性遗忘"和"分布外失效"的问题。

**阶段二：模块化端到端**
以UniAD为代表的方法，在端到端框架内部保留了感知-预测-规划的串行模块结构。这种方法既享受了端到端优化的好处，又维持了一定的可解释性。然而，它仍然需要完整的感知标注来训练各个模块。

**阶段三：生成式端到端**
近年来，扩散模型在图像生成、视频生成领域取得了革命性突破，这一技术也被引入端到端自动驾驶。DiffusionDrive、GoalFlow等工作将轨迹生成建模为去噪过程，能够生成多样化且合理的驾驶轨迹。但这些方法往往依赖预设的轨迹锚点（anchor）或目标点（goal point），限制了其灵活性。

### 2.2 感知标注的高昂代价

要理解TrajDiff的创新价值，我们必须首先正视感知标注的成本问题。以nuScenes数据集为例：

- **3D边界框标注**：每个场景包含数十个动态目标，每个目标需要精确的3D位置、尺寸、朝向标注
- **语义分割标注**：高清地图的标注需要在点云级别进行精细的语义分类
- **轨迹标注**：每个动态目标需要完整的时间序列轨迹
- **属性标注**：目标的速度、加速度、行为意图等

根据行业经验，一个专业的标注团队处理一个复杂城市场景的所有感知标注，平均需要**4-7小时**。考虑到现代自动驾驶数据集动辄包含数万个场景，总标注成本往往是**数百万美元**级别。

更令人担忧的是，这种高昂成本还带来了另一个问题：**数据多样性受限**。由于标注资源有限，数据集往往只能覆盖常见的驾驶场景，而对于各种边缘情况（corner cases）的覆盖则严重不足。这直接导致了自动驾驶系统在长尾场景下的脆弱性。

### 2.3 现有无标注方法的局限

针对上述问题，学术界已经开始探索无需感知标注的端到端方法。代表性的工作包括：

**Ego-MLP**：证明了仅使用自车状态（速度、加速度等）就能在开放回路（open-loop）设置下生成规划决策。但这显然忽略了环境感知，在实际应用中风险极高。

**SSR (Scene Representation)**：通过导航信号和简单的BEV预测来实现环境编码。但这种方法仍然依赖多帧传感器输入和复杂的预测模块。

**LAW (Learning from All Weathers)** 和 **World4Drive**：采用未来视觉隐变量预测和意图感知的隐变量预测来进行自监督场景表征学习。这些方法虽然摆脱了对感知标注的依赖，但仍需额外的架构来预测中间场景表征，引入了复杂性和潜在瓶颈。

更重要的是，**现有方法都没有充分利用未来轨迹来构建面向规划的自我监督目标**。而TrajDiff的核心洞见正是：**未来轨迹本身就蕴含了丰富的驾驶场景信息，我们可以从中提取自我监督信号，而无需任何人工标注**。

## 三、TrajDiff核心方法详解

### 3.1 整体架构概览

TrajDiff的整体架构由两大核心组件构成：

1. **Trajectory-oriented BEV Encoder（轨迹导向的BEV编码器）**：负责从原始传感器输入中提取面向轨迹的BEV特征（TrajBEV），通过高斯BEV热力图进行自监督训练
2. **Trajectory-oriented BEV Diffusion Transformer (TB-DiT)**：基于扩散模型的轨迹生成器，利用TrajBEV特征和自车状态来生成多样化的未来轨迹

整个系统的输入包括：
- **原始传感器数据**：多视角相机图像、LiDAR点云
- **自车状态**：当前速度、加速度、历史动作
- **导航指令**：高层驾驶意图（如"左转"、"直行"）

输出是**未来T秒内的轨迹点序列**，每个点包含位置和朝向信息。

### 3.2 高斯BEV热力图：从轨迹到监督信号

TrajDiff最核心的创新之一，就是提出了**高斯BEV热力图（Gaussian BEV Heatmap）**的概念，将未来轨迹转化为可用于自监督学习的空间目标。

**核心思想**：既然我们已经有未来轨迹的标注（在自动驾驶数据收集中，自车轨迹通常通过GPS/IMU自动记录，无需人工标注），为什么不从中提取空间信息来监督BEV特征的学习呢？

具体来说，对于每一个未来时刻t，自车的位置(x_t, y_t)被用作高斯分布的均值，而标准差则根据该时刻的速度自适应确定：

```
GT_xy = max_i (exp(-((x-x_i)² + (y-y_i)²) / (2*(Γ*v_i)²)))
```

其中：
- (x_i, y_i) 是第i个未来时刻的自车位置
- v_i 是该时刻的自车速度
- Γ 是控制速度影响半径的超参数

**为什么使用速度自适应的标准差？**

这个设计体现了作者对驾驶物理的深刻理解。当车辆高速行驶时，未来的可达空间是一个拉长的椭圆区域（考虑制动距离）；而当车辆低速行驶时，可达空间则更接近圆形。通过让标准差与速度成正比，高斯热力图能够准确刻画这种动态变化的可行驶区域。

图1展示了这一概念的直观效果。可以看到，高斯热力图不仅编码了未来的位置信息，还隐含了速度分布信息—— brighter的区域表示更有可能的驾驶路径。

### 3.3 Trajectory-oriented BEV Encoder

基于上述高斯热力图目标，TrajDiff设计了轻量级的轨迹导向BEV编码器，其核心组件包括：

**BEV热力图查询（BEV Heatmap Queries）**

不同于传统的BEV编码器直接预测感知目标（如3D边界框），TrajDiff初始化了一组可学习的BEV热力图查询 Q_heatmap ∈ R^{N×C}。这些查询通过Transformer编码器与BEV特征和自车状态特征进行交互：

```
Q̂_heatmap = G_H(Q_heatmap, Concat(F_bev, F_ego))
```

其中G_H是Transformer编码器，F_bev来自Transfuser编码器，F_ego是自车状态编码。

为了实现计算效率，这些BEV查询采用了相对于原始BEV网格的4倍下采样（即N = H×W/16）。

**热力图解码器**

Q̂_heatmap通过一个轻量级的上采样卷积网络D_H，生成单通道的BEV热力图：

```
F_heatmap = D_H(Q̂_heatmap),  F_heatmap ∈ R^{H×W×1}
```

这个热力图F_heatmap通过高斯Focal Loss进行监督学习：

```
L_bev = -Σ_{x,y} {
    (1-P_xy)^α log(P_xy)               if GT_xy = 1
    P_xy^α (1-GT_xy)^γ log(1-P_xy)     if GT_xy < 1
}
```

这种损失函数的设计借鉴了CenterNet等目标检测方法，对正样本（未来自车位置）附近的区域给予较小的惩罚，因为这些区域仍然是高置信度的可行驶区域。

**特征融合与TrajBEV**

F_bev和F_heatmap通过一个轻量级的通道融合网络G_fuse进行融合，生成最终的轨迹导向BEV特征（TrajBEV）：

```
F_traj = G_fuse(Concat(F_bev, F_heatmap))
```

TrajBEV是连接感知和规划的关键桥梁——它既保留了原始BEV特征的环境信息，又注入了从未来轨迹提取的驾驶模式信息。

### 3.4 Trajectory-oriented BEV Diffusion Transformer (TB-DiT)

TB-DiT是TrajDiff的第二个核心组件，负责基于TrajBEV特征生成未来的驾驶轨迹。这是一个基于扩散Transformer（Diffusion Transformer, DiT）的生成模型。

**扩散模型基础**

扩散模型包含两个过程：
1. **前向过程（Forward Process）**：逐步向原始数据添加高斯噪声
2. **反向过程（Reverse Process）**：学习从噪声中恢复原始数据

在TrajDiff中，前向过程将原始未来轨迹X_0通过T步变换为纯噪声：

```
q(X_t | X_0) = N(X_t; √ᾱ_t X_0, (1-ᾱ_t)I)
```

其中X_t是第t步的加噪样本，ᾱ_t是噪声调度参数。

通过重参数化技巧，我们可以直接采样：

```
X_t = √ᾱ_t X_0 + √(1-ᾱ_t) ε_t,  ε_t ~ N(0,I)
```

**噪声预测网络**

反向过程由一个噪声预测网络ε_θ来引导。给定当前的噪声样本X_t、时间步t和条件信息，网络预测所添加的噪声：

```
X_{t-1} = (1/√α_t)(X_t - (1-α_t)/√(1-ᾱ_t) ε_θ) + σ_t δ
```

其中δ ~ N(0,I)，σ_t是噪声水平。

**TB-DiT的架构设计**

TB-DiT的噪声预测网络采用了精心设计的架构，主要包含以下关键组件：

1. **Ego-BEV交互模块（G_EB）**

   这是一个跨注意力模块，用于增强自车查询与TrajBEV特征之间的交互：
   
   ```
   Q̂_ego = G_EB(Q_ego, F_traj)
   ```
   
   其中Q_ego是可学习的自车查询参数。这个模块的作用是将TrajBEV中编码的驾驶模式信息注入到自车查询中。

2. **条件编码**

   与标准DiT类似，TB-DiT使用扩散时间步嵌入F_t作为条件。此外，编码了驾驶模式信息的自车查询Q̂_ego也被用作额外的条件（类似于类别嵌入）：
   
   ```
   C = F_t + Q̂_ego
   ```
   
   这个复合条件C用于预测AdaLN-Zero的参数，对网络激活进行维度级的缩放和平移。

3. **噪声轨迹编码**

   输入的加噪轨迹X_t ∈ R^{T_f×3}通过可学习的轨迹编码器编码为隐变量Z_t ∈ R^{T_f×C}。

4. **TB-DiT块**

   每个TB-DiT块包含三个关键模块：
   
   - **时序自注意力（Temporal Self-Attention）**：沿轨迹的时间维度建模帧间交互
     ```
     Ẑ_t = SA_t(Z_t)
     ```
   
   - **BEV跨注意力（BEV Cross-Attention）**：建模轨迹与TrajBEV特征之间的交互
     ```
     Z̄_t = CA_BEV(Ẑ_t, Q_BEV, Q_BEV)
     ```
     这里使用了稀疏可学习查询和Q-Former结构来压缩高分辨率BEV特征。
   
   - **MLP层**：对处理后的特征进行精炼
     ```
     X̄_t = D_traj(Z̄_t)
     ```

**训练损失**

TB-DiT的训练使用标准的扩散损失：

```
L_Diffusion = E[||ε_θ(X_t, t, Q_ego, Q_BEV) - ε_t||²]
```

最终的损失函数是BEV损失和扩散损失的加权和：

```
L_final = ω_1 L_bev + ω_2 L_Diffusion
```

默认设置中，ω_1 = 200，ω_2 = 10。

### 3.5 无需锚点的轨迹生成

与DiffusionDrive等方法不同，TrajDiff采用**完全无锚点（anchor-free）**的设计。这意味着：

- 不需要预设的轨迹词汇表（trajectory vocabulary）
- 不需要预定义的轨迹锚点集合
- 不需要目标点（goal point）作为先验

这种设计的好处是多方面的：

1. **简化系统**：移除了锚点设计、聚类等预处理步骤
2. **提升灵活性**：模型可以自由探索整个轨迹空间，不受预设锚点的限制
3. **便于数据扩展**：没有锚点意味着新数据的引入不需要重新设计锚点集合

从实际效果来看，这种设计在NAVSIM基准上取得了优异的性能，证明了锚点并非端到端自动驾驶的必要条件。

## 四、实验结果深度分析

### 4.1 实验设置与基准

**NAVSIM基准**

TrajDiff在NAVSIM基准上进行训练和评估。NAVSIM是一个基于真实驾驶数据的高挑战性端到端自动驾驶基准，具有以下特点：

- **数据来源**：基于OpenScene数据集，包含真实的城市场景
- **传感器配置**：8个摄像头视角 + 5个LiDAR传感器融合的点云
- **预测要求**：预测未来4秒内的8个轨迹点
- **评估方式**：非反应式仿真（non-reactive simulation）+ 闭环指标

**数据集划分**：
- navtrain：1192个场景（978个训练 + 214个验证）
- navtest：136个测试场景

值得注意的是，虽然NAVSIM提供了2Hz的物体和地图标注用于基于感知的方法训练，但这些标注对TrajDiff等无标注方法是不可见的。

**评估指标**

NAVSIM使用**预测驾驶模型评分（PDMS）**作为主要评估指标，这是一个由五个子指标加权组合的复合指标：

1. **NC (No at-fault Collisions)**：无责任碰撞率
2. **DAC (Drivable Area Compliance)**：可行驶区域遵守率
3. **TTC (Time-To-Collision)**：碰撞时间
4. **Comf. (Comfort)**：舒适性
5. **EP (Ego Progress)**：自车进度

### 4.3 消融实验：验证每个设计的价值

**TB-DiT设计组件的有效性**

表2展示了逐步添加各个组件的效果：

| 配置 | PDMS |
|------|------|
| 无感知信息（基线） | 81.2 |

4. **Ego-BEV交互模块贡献0.2 PDMS提升**，BEV跨注意力贡献0.4 PDMS提升**，每个模块都是不可或缺的。

| 配置 | PDMS | NC | DAC | TTC | Comf. | EP |
|------|------|-----|-----|-----|-------|-----|
| 恒定半径（大） | 85.9 | 97.4 | 95.5 | 91.3 | 100 | 78.8 |
| 恒定半径（小） | 87.2 | 97.9 | 96.9 | 93.8 | 100 | 82.1 |
| **速度自适应** | **87.5** | **98.1** | **97.0** | **94.3** | **100** | **82.7** |

结果表明：

- **过大的恒定半径导致性能显著下降**（特别是DAC和EP指标），因为过度扩大的可行驶区域会让模型产生不切实际的预测。

- **较小的恒定半径能生成合理轨迹**，但会强制TrajBEV过度关注特定位置。

- **速度自适应目标在NC和TTC上有边际但一致的改进**，证明其增强了环境感知能力。

### 4.4 数据扩展：释放无标注范式的潜力

TrajDiff的一个重要优势是**能够直接从数据扩展中受益**，而不受标注成本的限制。

表3展示了数据扩展的效果：

| 配置 | PDMS |
|------|------|
| 基线（978场景） | 87.5 |
| + 初始点变化重采样 | 87.9 |
| + 使用完整navtrain（1192场景） | 88.1 |
| **两者结合** | **88.5** |

这里使用的数据扩展策略非常简单：

1. **初始点变化重采样**：通过扰动轨迹的起始点来生成额外的训练样本
2. **使用完整数据集**：将验证集的214个场景也纳入训练

关键洞见：

- **即使没有额外的场景多样性，仅仅通过轨迹数量扩充就能提升性能**
- **这说明TrajDiff能够从未经标注的数据中提取有意义的模式**
- **无标注范式的可扩展性潜力得到了充分验证**

相比之下，基于感知标注的方法如果要进行同样的数据扩展，需要额外标注214个场景，成本高达**约1500人工小时**。这种成本差异在大规模数据扩展时将变得更加显著。

### 4.5 轨迹多样性与采样分析

扩散模型的一个核心优势是能够生成多样化的样本。TrajDiff通过并行采样K条轨迹，并选择最佳轨迹，可以进一步提升性能。

表5展示了不同采样数量K的效果：

| K | PDMS | 标准差均值 |
|---|------|-----------|
| 1 | 87.5 | 0 |
| 3 | 89.8 | 0.42 |
| 5 | 90.8 | 0.51 |
| 10 | **92.0** | 0.64 |

结果表明：

- **随着K增加，PDMS显著提升**，说明更大的采样候选集更有可能包含更优解
- **10次采样时，TrajDiff达到92.0 PDMS**，创造了新的性能记录
- **标准差随K增加而逐渐增加**，验证了采样结果的多样性

这种多样性对于处理复杂和不确定的驾驶场景至关重要。当面对模糊的情况（如前方车辆可能变道也可能不变道）时，能够生成多条合理但不同的候选轨迹，然后选择最优的一条，是提升系统鲁棒性的关键。

### 4.6 实时性分析

表6对比了不同去噪步数下的性能与推理速度：

| 方法 | 采样步数 | PDMS | FPS | 参数量 |
|------|---------|------|-----|--------|
| DiffusionDrive | 20 | 87.9 | 2.4 | 62M |

- **当FPS提升至9.8（5步采样）时，PDMS仅下降至86.9**，仍然具有很强的竞争力

这种效率优势来源于TrajDiff的简洁架构——没有锚点预测和解码的额外开销。

## 五、技术洞察与深度理解

### 5.1 为什么高斯热力图有效？

高斯BEV热力图的设计看似简单，实则蕴含深刻的驾驶物理洞察：

**1. 编码可达性约束**

驾驶本质上是一个受物理约束的运动规划问题。车辆的未来位置不是任意的，而是受限于：
- 当前状态（位置、速度、朝向）
- 动力学约束（最大加速度、转向角限制）
- 时间预算（预测_horizon）

高斯热力图通过将未来轨迹点转化为空间概率分布，自然编码了这些约束。 brighter的区域代表"可达"，dark的区域代表"不可达"。

**2. 速度自适应的合理性**

标准差与速度成正比的设定，符合驾驶的直观理解：
- 高速时：制动距离长，未来位置的不确定性范围大
- 低速时：机动性强，未来位置的不确定性范围小

这种设计让模型能够学习到"速度-可达区域"的映射关系，从而在规划时自然地考虑速度因素。

**3. 多峰性的隐式处理**

当面对多车道或复杂交叉口时，未来轨迹可能存在多个合理的选择（如左转或右转）。高斯热力图通过在每个未来时刻放置一个高斯分布，并通过max操作聚合，隐式地处理了这种多峰性。

### 5.2 无锚点设计的深层优势

传统的基于锚点的方法存在几个固有问题：

**1. 锚点设计的困境**

- 锚点太少：覆盖不足，可能遗漏最优轨迹
- 锚点太多：计算开销大，且可能引入噪声
- 锚点分布：如何在动作空间中均匀分布锚点本身就是一个难题

**2. 数据偏差的放大**

锚点通常通过聚类训练数据获得，这会放大数据集中的偏差。如果训练数据中某种驾驶行为较少（如紧急避让），对应的锚点也会较少，导致模型在这类场景下表现不佳。

**3. 分布外泛化的困难**

当遇到训练分布之外的场景时，预设的锚点可能完全失效。而无锚点方法由于没有预设约束，理论上具有更好的分布外泛化能力。

TrajDiff的无锚点扩散生成方式，让模型直接在连续的轨迹空间中进行优化，避免了上述问题。

### 5.3 无标注 vs 有标注：一场范式革命？

TrajDiff的结果引发了关于自动驾驶训练范式的深层思考：**我们真的需要感知标注吗？**

**支持无标注范式的论据**：

1. **目标对齐**：端到端自动驾驶的最终目标是生成好的轨迹，而不是精确地检测每一个障碍物。如果模型能从轨迹中直接学到足够的驾驶知识，中间表征的精确性可能并非必要。

2. **数据规模的力量**：TrajDiff的数据扩展实验表明，即使没有更多场景多样性，仅仅增加数据量就能带来性能提升。这意味着**数据规模可能比标注精度更重要**。

3. **长尾覆盖**：无标注方法可以更容易地扩展到罕见场景，因为只需要收集数据，不需要昂贵的标注。

**保留有标注方法的论据**：

1. **可解释性**：感知标注提供了对模型决策过程的可解释性，这对安全关键系统至关重要。

2. **故障诊断**：当模型出错时，中间感知结果可以帮助定位问题根源。

3. **多任务迁移**：感知表征可以迁移到其他任务（如预测、模拟）。

**可能的融合方向**：

未来的自动驾驶系统可能采用"双轨制"：
- **主要训练**：使用无标注方法进行大规模预训练，充分利用未标注数据
- **精调与验证**：使用少量标注数据进行监督精调和安全验证

这种模式既能享受数据规模的红利，又能保留必要的可解释性和安全性保证。

## 六、对行业的影响与展望

### 6.1 对自动驾驶研发的直接影响

**降低数据成本**

TrajDiff最直接的影响是大幅降低了自动驾驶研发的门槛。对于初创公司和研究机构来说，不再需要投入数百万美元进行数据标注，可以将资源投入到算法创新和系统优化上。

**加速数据闭环**

无标注训练使得"收集数据→训练模型→部署测试→收集更多数据"的闭环更加高效。新收集的数据可以立即用于训练，无需等待漫长的标注过程。

**长尾场景覆盖**

自动驾驶的安全瓶颈往往在于罕见的长尾场景。TrajDiff使得收集和训练这些场景数据变得更加可行——不再需要为每个罕见场景标注大量感知数据。

### 6.2 技术发展趋势预测

**趋势一：感知与规划的进一步融合**

TrajDiff代表了"感知-规划一体化"的极端形态——甚至不需要明确的感知输出。未来我们可能看到更多中间形态：

- 弱监督感知：使用粗粒度标注（如场景级别标签）而非细粒度标注
- 自监督感知：从时空一致性、几何约束等信号中学习感知能力
- 隐式感知：像TrajDiff这样，感知能力隐式编码在规划模型中

**趋势二：生成式模型的主导地位**

扩散模型在TrajDiff中的成功，加上VADv2、DiffusionDrive、GoalFlow等工作的积累，表明**生成式模型正在成为端到端自动驾驶的主流范式**。其优势在于：

- 天然处理多模态不确定性
- 能够生成多样化的候选方案
- 与数据扩展天然契合

**趋势三：世界模型的崛起**

TrajDiff的数据扩展实验揭示了数据规模的重要性。但要真正发挥大数据的威力，我们需要能够理解和预测世界动态的世界模型。WoTE、World4Drive等工作已经在这个方向上进行了探索，未来这一趋势将加速。

### 6.3 对自动驾驶行业的长远影响

**商业模式的变革**

如果无标注训练成为主流，自动驾驶行业的商业模式可能发生根本性变化：

- **数据优势将重新分配**：拥有大规模车辆部署能力的公司（如特斯拉、Waymo）将获得更大的数据优势
- **标注服务提供商面临转型**：纯标注业务可能萎缩，转向更高端的验证和精调服务
- **开源生态繁荣**：更多研究者和开发者能够参与自动驾驶创新

**技术路线的不确定性**

TrajDiff的成功也让我们重新思考自动驾驶的技术路线：

- **模块化的必要性**：如果端到端模型能够直接从数据中学习驾驶，复杂的多模块架构是否还有必要？
- **规则与学习的平衡**：完全数据驱动的方法在安全关键场景下是否足够可靠？
- **可解释性的代价**：为了可解释性而保留的模块化设计，是否值得牺牲性能和可扩展性？

这些问题没有标准答案，但TrajDiff为我们提供了一个重要的参考点：至少在某些场景下，无标注的端到端学习是可行的，而且效果可能出乎意料地好。

## 七、局限性与未来工作

### 7.1 当前局限

尽管TrajDiff取得了令人瞩目的成果，但仍存在一些局限性：

**1. 开环评估的限制**

NAVSIM采用的是非反应式仿真评估，即周围环境对自车行为没有反应。这与真实世界的闭环场景存在差距。TrajDiff在真实闭环场景下的表现还有待验证。

**2. 极端场景的挑战**

虽然TrajDiff在标准测试集上表现优异，但在极端场景（如极端天气、传感器故障、其他交通参与者的异常行为）下的鲁棒性尚未充分验证。

**3. 多模态融合的深度**

TrajDiff使用相对简单的Transfuser编码器来融合相机和LiDAR数据。更先进的多模态融合技术（如基于注意力的深度融合、自适应模态选择）可能进一步提升性能。

**4. 长期规划能力**

TrajDiff预测4秒内的轨迹。对于更长horizon的规划（如高速场景需要预测8-10秒），当前方法可能需要调整。

### 7.2 潜在改进方向

**1. 与强化学习的结合**

TrajDiff使用模仿学习进行训练。结合强化学习（如DiffusionDrive使用的RLHF）可能进一步提升在复杂交互场景下的表现。

**2. 在线适应与终身学习**

如何让TrajDiff在部署后持续学习，适应新的场景和驾驶风格，是一个重要的研究方向。

**3. 不确定性量化**

当前的扩散采样提供了多样性，但缺乏对每条轨迹置信度的量化。显式的不确定性估计对安全决策至关重要。

**4. 多智能体场景**

TrajDiff主要关注自车规划。在多智能体交互场景下，如何联合建模其他交通参与者的行为，是未来的重要挑战。

## 八、结语

TrajDiff的提出，标志着端到端自动驾驶进入了一个新阶段——**无感知标注学习**的阶段。这项工作不仅在技术上取得了突破（87.5 PDMS的SOTA性能），更重要的是，它挑战了我们对自动驾驶训练范式的固有认知。

这篇论文的核心启示在于：**在端到端框架下，未来轨迹本身就蕴含了足够的监督信号**。通过巧妙的高斯BEV热力图设计，TrajDiff将轨迹标注转化为空间监督目标，实现了完全自监督的BEV特征学习。配合无锚点的扩散生成框架，整个系统简洁、高效、可扩展。

对于自动驾驶行业而言，TrajDiff的影响可能是深远的。它降低了数据门槛，让更多人能够参与自动驾驶的研发；它提升了数据效率，让大规模数据扩展成为可能；它挑战了模块化范式，为端到端架构的进一步简化提供了思路。

当然，TrajDiff不是终点。在通往完全自动驾驶的道路上，我们还有很长的路要走。但这篇论文至少告诉我们：**有时候，解决问题的方法不是增加更多的标注，而是重新思考问题的本质**。

在数据和算力日益成为核心竞争力的今天，TrajDiff为我们指明了一条更可持续的道路——用更聪明的方法，而不是更多的资源，来解决复杂的问题。这或许正是人工智能领域最宝贵的洞见。
